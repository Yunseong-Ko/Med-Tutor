# Axioma Qbank 성능 최적화 우선순위 로드맵

## 1. 목표
- 가장 중요한 제품 지표를 `문제 생성 완료까지 걸리는 시간`과 `페이지 전환 체감 속도`로 둔다.
- 기능 추가보다 먼저, 사용자가 "기다릴 만하다"가 아니라 "바로 쓸 수 있다"라고 느끼게 만드는 것을 우선한다.

## 2. 현재 코드 기준 병목 요약
현재 구조를 보면 병목은 크게 4군데다.

### A. 생성 전처리 병목
- 업로드 후 텍스트 추출, OCR/AI fallback, 스타일 파일 파싱이 Streamlit 요청 흐름 안에서 수행된다.
- 파일 수가 많거나 스캔 PDF가 많으면 대기열 추가 단계 자체가 길어진다.

### B. 생성 완료 후 저장/리렌더 병목
- 생성 큐 상태 변경 때마다 전체 페이지가 다시 렌더링된다.
- 같은 페이지 안에서 대기열, 통계, 기타 위젯이 함께 다시 그려져 체감이 무거워진다.

### C. 홈/시험 페이지 렌더 병목
- 질문 수가 늘어날수록 `load_questions()` 이후 집계, 히트맵, 표 렌더가 반복된다.
- Streamlit 특성상 작은 상호작용도 전체 스크립트 재실행으로 이어져 누적 비용이 커진다.

### D. 캐시 지속성 부족
- 현재는 세션 캐시 중심이라 브라우저 세션이 바뀌거나 배포 환경이 재시작되면 전처리 이점을 충분히 못 살린다.

## 3. 우선순위

### P0. 가장 먼저 할 것
#### 1) 파일 해시 기반 영속 전처리 캐시
- 키: `file_hash + parser_mode + ocr_mode + ai_model`
- 저장 대상:
  - 추출 텍스트
  - 페이지 수
  - 예상 청크 수
  - 추출 실패 여부
- 효과:
  - 같은 강의록 재업로드 시 거의 즉시 대기열 추가 가능
  - 베타테스터가 여러 번 같은 자료를 돌려도 반복 비용 감소

#### 2) 홈 화면 무거운 집계 lazy-load
- 히트맵, 전체 통계, 오답 분석을 기본 즉시 계산하지 않고:
  - 첫 화면은 핵심 요약만 표시
  - 상세 분석은 expander 또는 버튼 눌렀을 때 계산
- 효과:
  - 로그인 직후/페이지 이동 직후 체감 속도 개선

#### 3) 질문 데이터 집계 1회화
- 한 페이지 안에서 `load_questions()`와 파생 계산을 여러 번 하지 않도록 페이지 단위 캐시 객체로 묶기
- 예:
  - `question_bank`
  - `all_questions`
  - `subject_unit_map`
  - `stats`
  - `wrong_items`
- 효과:
  - 렌더 단계 중복 계산 감소

#### 4) 생성 큐 UI 리렌더 범위 축소
- 생성 큐 상태 업데이트 시:
  - 전체 페이지가 아니라 큐 관련 섹션만 변화하는 구조로 점진 분리
- Streamlit 특성상 완전한 부분 렌더는 어렵지만, 무거운 섹션을 페이지 하단/expander로 밀어 영향 감소 가능

## 4. P1. 그 다음 단계

### 1) 백그라운드 워커 분리
- 현재 생성과 추출이 Streamlit 프로세스에 붙어 있어 UI 응답성과 경쟁한다.
- 추천 구조:
  - 웹(UI): Streamlit
  - 작업자(worker): 별도 Python process / queue consumer
  - 저장소: Supabase + local cache or file cache
- 효과:
  - 생성 중에도 페이지 이동, 기록 확인, 시험 풀이가 훨씬 안정적

### 2) 생성 결과 부분 저장
- 지금은 한 번에 전부 끝난 뒤 저장한다.
- 청크별 완료 시 미리 저장하거나 미리보기 가능하게 바꾸면 사용자는 더 빨리 결과를 체감한다.

### 3) 스타일 프로파일 캐시
- 기출 스타일 파일도 자주 재사용되므로:
  - 용어 표기
  - 난이도
  - 문제 유형 분포
  - 문장 길이
  를 별도 profile로 캐싱

## 5. P2. 이후 단계

### 1) 검색/인덱싱 기반 생성 준비
- 큰 강의록 전체를 매번 그대로 던지지 않고:
  - 단원 인덱싱
  - 핵심 문단 추출
  - 이미지/표 위치 태깅
  후 필요한 부분만 생성에 사용

### 2) 사용성 기준 자동 프리셋
- 파일 크기/페이지 수를 보고 자동 제안:
  - `빠름`: 5~10문항
  - `균형`: 10~15문항
  - `정밀`: 15~25문항

## 6. 이번 달 구현 권장 순서
1. 영속 전처리 캐시
2. 홈 화면 lazy-load
3. 페이지 단위 질문 집계 캐시
4. 생성 큐/생성 페이지 UI 분리
5. 백그라운드 워커 설계 시작

## 7. 측정 지표
- `t_upload_to_queue_ms`
- `t_queue_to_first_done_ms`
- `t_total_generation_ms`
- `home_page_render_ms`
- `exam_page_render_ms`
- 캐시 히트율:
  - `extract_cache_hit`
  - `style_cache_hit`

## 8. 목표 수치
- 홈 화면 첫 진입: 1.5초 이내
- 페이지 전환: 1초 내외
- 동일 파일 재생성 시 대기열 추가: 3초 이내
- 10문항 생성(텍스트 PDF): 1~3분대
- 20문항 이상 생성 시에도 사용자가 단계별 진행 상황을 볼 수 있어야 함

## 9. 결론
현 시점에서 가장 ROI가 큰 작업은 `모델을 바꾸는 것`이 아니라 `전처리 캐시 + 렌더 lazy-load + 생성/저장 파이프라인 분리`다.  
이 3가지를 먼저 잡으면 베타 사용자 입장에서 가장 크게 느끼는 "느리다" 문제가 가장 빠르게 줄어든다.
