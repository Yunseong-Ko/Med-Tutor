# Axioma Qbank 운영 계획표

## 1. 운영 목표
- 1순위: 문제 생성 시간과 사이트 체감 속도 개선
- 2순위: 생성된 문제를 반복 학습 가능한 Qbank UX로 고도화
- 3순위: 스타일 적합도와 기관 확장성을 준비

## 2. 운영 원칙
- `속도 개선`이 새 기능보다 우선
- 각 주차는 `기획 -> 구현 -> 검증 -> 베타 반영`까지 닫힌 루프로 운영
- 성능 작업은 반드시 수치 지표를 남김
- 베타 유저 피드백은 기능 요청보다 `생성 성공률`, `대기시간`, `반복 사용 여부` 중심으로 정리

## 3. 8주 운영 계획

| 주차 | 목표 | 핵심 작업 | 산출물 | 검증 기준 |
|---|---|---|---|---|
| 1주차 | 생성 속도 측정 기반 확보 | 생성/렌더 시간 계측 추가, 병목 구간 로그화 | 성능 로그 지표 정의서 | `t_upload_to_queue`, `t_generate`, `t_render` 수집 가능 |
| 2주차 | 전처리 반복 비용 절감 | 파일 해시 기반 영속 전처리 캐시 구현 | 캐시 설계 + 동작 구현 | 같은 파일 재생성 시 전처리 재호출 감소 |
| 3주차 | 페이지 체감 속도 개선 | 홈/시험 페이지 lazy-load, 무거운 집계 지연 계산 | 렌더 최적화 반영본 | 홈 진입 체감 속도 개선, 초기 렌더 단축 |
| 4주차 | Qbank 반복 사용성 강화 | Saved Session Templates, 문제 상태 필터 설계/구현 | 템플릿/상태 필터 V1 | 사용자가 기존 문제만으로 세션 구성 가능 |
| 5주차 | 리뷰 경험 강화 | Review-First UX, 오답/북마크/메모 흐름 통합 | 시험 종료 후 리뷰 흐름 개선 | 리뷰 진입률 증가, 오답 재풀이 경로 단축 |
| 6주차 | 스타일 적합도 준비 | Style Profile schema 설계, 품질 피드백 수집 버튼 추가 | 프로파일 스키마 초안 | 강의/기출 스타일 저장 구조 확정 |
| 7주차 | 기관 기능 사전 설계 | Assignment Mode, cohort analytics 요구사항 정의 | 교육실용 기능 요구사항 문서 | 기관 기능 우선순위 정리 완료 |
| 8주차 | 구조 안정화 | 워커 분리 초안, 모듈화 지속, 회귀 테스트 보강 | 기술 부채 정리 리포트 | 다음 분기 확장 준비 완료 |

## 4. 각 단계별 운영 상세

### Week 1-2: 성능 측정과 캐시
#### 핵심 질문
- 생성 시간은 실제로 어디서 가장 많이 쓰이는가?
- 같은 파일을 다시 넣을 때 얼마나 반복 낭비가 발생하는가?

#### 실행 항목
- 생성 시작/완료/저장 시간을 감사 로그와 별도로 추적
- 추출 텍스트, 스타일 파싱, OCR fallback 히트율 기록
- 파일 해시 캐시 도입

#### 기대효과
- "느리다"를 느낌이 아니라 데이터로 볼 수 있게 됨
- 같은 강의록 재업로드 시 체감 속도 큰 폭 개선

### Week 3: 렌더 최적화
#### 핵심 질문
- 어떤 페이지가 가장 무거운가?
- 어떤 위젯이 전체 rerun 비용을 키우는가?

#### 실행 항목
- 홈의 상세 통계/히트맵 lazy-load
- 시험 페이지의 집계/필터 결과 캐시
- 무거운 표/차트는 요청 시 계산

#### 기대효과
- 로그인 직후/페이지 이동 직후 체감 속도 개선

### Week 4-5: Qbank UX 강화
#### 핵심 질문
- 사용자가 문제를 한 번 생성한 뒤 다시 돌아오게 만드는 요소는 무엇인가?

#### 실행 항목
- 세션 템플릿 저장
- 북마크 / 오답 / 미풀이 / 최근 생성 필터
- 리뷰 우선 흐름
- 메모와 오답 복습 연결

#### 기대효과
- 생성 기능 의존도만 높은 툴에서, 반복 학습 가능한 문제은행으로 진화

### Week 6: 스타일 프로파일 준비
#### 핵심 질문
- "우리 학교 스타일 같다"를 어떻게 구조화할 것인가?

#### 실행 항목
- 용어 표기, 문항 길이, Basic/Case 비율, 자주 묻는 유형을 profile schema로 정의
- 사용자 피드백 수집 버튼 추가

#### 기대효과
- Axioma의 핵심 차별점인 스타일 적합도를 제품적으로 다룰 수 있음

### Week 7-8: 기관 확장과 구조 안정화
#### 핵심 질문
- 교육실/교수 관점에서 꼭 필요한 최소 기능은 무엇인가?
- 지금 구조로 기관용 확장이 가능한가?

#### 실행 항목
- Assignment Mode 요구사항
- cohort analytics 요구사항
- 백그라운드 worker 분리 설계
- 모듈화와 테스트 확장

#### 기대효과
- B2C 학습 도구에서 B2B 확장 가능한 구조로 이동

## 5. 운영 지표

### 제품 지표
- 생성 성공률
- 평균 생성 완료 시간
- 세션 재방문율
- 저장된 문제 재사용률
- 시험 후 리뷰 진입률

### 기술 지표
- `t_upload_to_queue_ms`
- `t_generate_total_ms`
- `home_render_ms`
- `exam_render_ms`
- `extract_cache_hit_rate`
- `style_cache_hit_rate`

## 6. 릴리즈 정책
- 매주 1회 브랜치 기준 통합 점검
- `속도 개선` 작업은 반드시 테스트 + 수동 검증 후 반영
- 베타 사용자 노출 전 체크:
  - 생성 성공
  - 중복 저장 없음
  - 삭제/내보내기 회귀 없음
  - 로그인/계정 분리 정상

## 7. 외부 벤치마크를 더 깊게 봐야 하는 시점
디버깅 모드나 추가 리서치가 특히 유효한 시점은 아래다.

1. `Week 4-5`
- UWorld / AMBOSS의 리뷰 UX, 해설 구성, 필터 구조를 더 깊게 비교할 때

2. `Week 6`
- 스타일 적합도와 AI 힌트 구조를 설계할 때

3. `Week 7`
- 교육실/기관 기능을 붙일 때 다른 교육 플랫폼의 관리형 기능을 비교할 때

즉, 지금 당장은 성능 최적화 구현이 먼저고, Qbank 벤치마크의 심화 탐구는 `Week 4`부터 붙이는 것이 가장 효율적이다.

## 8. 이번 주 바로 실행할 것
- 성능 로그 지표 추가
- 전처리 캐시 구현 설계 시작
- 홈/시험 lazy-load 후보 목록 정리
- 베타 사용자 피드백을 `속도/생성 성공/반복 사용` 기준으로 재분류

## 9. 결론
운영 관점에서 가장 중요한 것은 기능 수를 늘리는 것이 아니라,  
`느리지 않게 만들고`, `이미 만든 문제를 계속 쓰게 만들고`, `그 다음에 개인화와 기관 확장으로 넘어가는 것`이다.

따라서 이번 8주 운영의 핵심은 한 줄로 정리된다.

**생성형 문제은행을 빠르고 반복 사용 가능한 제품으로 안정화한다.**
